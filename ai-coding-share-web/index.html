AI Coding
不只是写代码

把散碎技巧整理成一个最小流程：任务写清楚，边界说清楚，验证交出来，记录留得住。

> prompt   : 工单化
> context  : 项目说明书

> trace    : 版本记录接住操作

01 / problem

为什么 AI Coding 经常跑不稳

模糊需求AI 只能猜，猜对靠运气。
缺少上下文不知道项目边界和禁区。
没有验收看似完成，但没人知道对不对。
没有留痕改了什么、为什么改、怎么回滚都说不清。

02 / prompt

Prompt 要像工单，不像聊天

一条适合 AI 执行的任务，至少要包含背景、目标、输入、范围、限制、输出和验收。

task.issue 
背景：减少入版前重复沟通
目标：整理一次活动入口入版任务包
限制：不修改现有代码脚本
输出：工单 / 材料清单 / 验收 checklist
验收：列出已确认、未验证、需人工判断的部分

03 / plan

先让 AI 计划，再让 AI 执行

先不要写代码。

先复述理解。

先列实现计划。

先暴露风险点。

先提出需要确认的问题。

计划不是拖慢效率，而是在自动化之前先暴露缺口。

04 / context

给 AI 一份项目说明书

稳定的自动化不能只靠临时 Prompt。目录结构、命令、风格、禁区和验证方式，需要有长期入口。

AGENTS.md CLAUDE.md .github/copilot-instructions.md team-rules.md

05 / automation

从低风险小闭环开始

优先自动化

  • 任务拆解
  • 报告生成
  • 验收清单
  • 变更说明

先别直接放开

  • 真实项目资产自动修改
  • 高风险逻辑改动
  • 无法自动验证的修改
  • 体验和风险最终判断

06 / example

小示例：活动入口入版任务包

不让 AI 直接改正式项目资产。先把一次 Unity UI 入版需求整理成 AI 能参与的流程包。

input 
模块：活动入口面板

改动：
1. 新增限时活动入口
2. 替换入口 Banner
3. 增加倒计时展示
4. 增加红点状态
5. 点击入口跳转详情页

限制：
- 不修改活动开启条件逻辑
- 不修改红点系统底层逻辑
- 不改动项目内现有代码脚本
output 
输出：
1. AI 可执行工单
2. 入版前材料清单
3. 自动化可处理项
4. 人工确认点
5. 验收 checklist
6. P4 提交说明草稿

07 / takeaway

最后一句话

AI Coding 的重点不是让 AI 替我们写代码，而是把实际工作拆成 AI 能理解、能执行、能验证、能留下记录的流程。
